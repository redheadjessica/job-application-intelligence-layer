# AI Job-Search Pipeline

An open-source, run-it-locally pipeline that turns a messy job search into one chained flow: **source jobs → vet a batch → tailor the top few → learn from what you submit.**

Clone this repo, run **`/intake`** once to generate your own personal files from your resume, then run the pipeline on batches of job URLs. Everything personal to you is generated by intake; the engine is shared and stays generic.

It is organized as a four-stage pipeline — **Prep → Vet → Tailor → Reconcile**:

- **`prep/`** — **Prep.** Fetch each job URL into a clean job `.txt`: `prep_job_urls.py`, the Playwright fallback `prep_job_urls_playwright.py`, and the ATS-aware `ats_fetchers.py` (Greenhouse / Lever / Ashby / Workday / LinkedIn-guest). PII-free; no dependency on your profile.
- **`vetting/`** — **Vet.** Triage only: score and rank which jobs are worth applying to, using your scoring card + candidate profile. Writes a ranked CSV + Markdown + a formatted XLSX. See `vetting/CLAUDE.md`.
- **`tailor/`** — **Tailor.** Resume *tailoring* only (never submitting). Takes a target job and writes an `application_resume_output - [Company] - [Role].md` draft: picks a resume base, flags gaps, suggests content. See `tailor/00-job_application_agent.md`.
- **`tailor/learning/`** — **Reconcile.** Post-submission learning loop (maintenance-only, never read during a generation run): compares what the system recommended against what you actually submitted and proposes lessons for your review. `reconcile-spec.md`, `learning-ledger.md`, `source-update-queue.md`.

## Setup — run this first

Run **`/intake`** before anything else. It reads your resume(s), asks a few targeted questions, and generates your personal files:

- `vetting/01-scoring-card.md` — your rubric (4 weighted dimensions + 1–2 custom factors)
- `vetting/02-candidate-profile.md` — who you are, your priority lanes, practical constraints
- `tailor/01-profile.md`, `02-resume-index.md`, `03-current-work-canonical.md` (optional), `04-experience-bank.md`, `05a-summary-library.md` (grows over time), `10-bio-library.md` (optional)

Until intake has run, these files are blank templates and the pipeline has nothing to score or tailor against. Your resume base files live in **your own** resume folder (you point intake at it) — they are not stored in this repo.

## How to run (three modes)

First scaffold + fetch a batch: `python vetting/new_batch.py <MM-DD-YY>`, then run the fetch commands it prints. Then pick a mode (`<batch>` = `__READY TO REVIEW/<MM-DD-YY>`):

1. **Vet only (default):** `run-batch {folder: "<batch>"}` — vets and ranks into `1 - Rankings/`, then stops. You review the rankings and decide what to pursue.
2. **Vet + tailor top N:** `run-batch {folder: "<batch>", tailor: true, topN: 3}` — vets, then tailors resume drafts into `2 - Tailored Resumes/` for the top N (sequential, highest first, Skip-status excluded).
3. **Tailor a hand-picked set:** `tailor-jobs {jobs: ["<batch>/3 - Source Material/All Job Posts (full text)/jobA.txt", ...]}` — only the chosen jobs, in order. Use after a vet-only run.

Tailoring always runs the `job-applier` agent in autonomous mode: it never blocks on questions and puts an **"Open Questions"** section at the top of each output for you to resolve.

### Where a run writes

Everything a run produces lands in one place: **`__READY TO REVIEW/<batch>/`**, in three numbered tiers so the review order is top-down by importance — `1 - Rankings/`, `2 - Tailored Resumes/` (one `Company - Role` folder per tailored job), and `3 - Source Material/` (the fetched job `.txt` files + a snapshot of that batch's URLs). The leading `__` pins the folder to the top of the repo. Nothing is moved after the fact; `vetting/` and `tailor/` hold only engine code + your templates, never run outputs.

## What's yours vs. shared

- **Yours** (generated by `/intake`; hold your career, claims, and preferences — keep private if you fork publicly): everything in `vetting/0*.md` and `tailor/0*.md` plus `tailor/10-bio-library.md`.
- **Shared engine** (don't personalize): `prep/*`, `.claude/workflows/*`, `.claude/agents/*`, `tailor/00-job_application_agent.md`, the `05`/`06` summary/skills template files, `vetting/new_batch.py`, `vetting/make_rankings_xlsx.py`.

---

## Scope guardrail

This pipeline prepares drafts and organizes folders. It must **never submit applications** — that step is outward-facing and irreversible, and stays a human decision.

---

## Working style for this repo

A local, git-tracked workspace — routine file changes are reversible, so optimize for momentum: read, search, and edit freely; don't pause to ask "should I continue?" after routine steps. Stop and ask first only when an action reaches outside the repo, needs secrets/credentials/payment, is truly destructive and hard to undo (`rm -rf`, force-push over shared history), or changes dependencies/CI.

---

## Known assumptions (v1)

- **Resume bases** can be Word (`.docx`), PDF, plain text/Markdown, or Apple Pages — point intake at whatever you use (Google Docs: export to `.docx`/PDF first). Multi-format handling is being wired through the Tailor step; `.pages` is the most-tested path today.
- Python parts (`new_batch.py`, `make_rankings_xlsx.py`, `prep/`) need a local `.venv` with the deps in `requirements.txt` (notably `openpyxl` for the XLSX, and Playwright for the JS-rendered fetch fallback).
